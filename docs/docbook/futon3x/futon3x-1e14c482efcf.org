* Context
The three-column stack (docbook://futon3x/futon3x-5b802804c69b) stores math,
project, and code data as hyperedges. The invariants detect when these columns
disagree. This entry documents how the invariant checking works and how to
extend it.

* How invariants work

Each invariant is a query over the hyperedge store that returns a set of
/violations/. A violation is an entity that fails a structural property.

The check cycle:
1. Query futon1a for hyperedges of each relevant type
2. Compute set operations (e.g., "namespaces without doc" = all namespaces minus those with doc)
3. Write violations as hyperedges of type =invariant/<violation-type>=
4. Violations are browsable in Arxana and queryable via the futon1a API

* Current invariants

** INV-1: Undocumented Entry Points (Project+Code)

Every namespace in the code column should have a docstring. Undocumented
namespaces are entry points that the project column cannot describe.

#+BEGIN_SRC sh
# Query violations
curl -s "http://localhost:7071/api/alpha/hyperedges?type=invariant/undocumented-entry-point&limit=100"
#+END_SRC

/Resolution:/ Add a docstring to the namespace's =ns= form.

** INV-2: Uncovered Components (Project)

Every devmap component should have a covering mission. Uncovered components
are project-column gaps — things the system aspires to but hasn't started.

/Resolution:/ Create a mission or annotate coverage explicitly.

** INV-3: Orphan Namespaces (Code+Code)

Every namespace in the futon ecosystem should be required by at least one
other namespace (excluding known entry points like =dev= and =system=).
Orphans may be dead code.

/Resolution:/ Verify the namespace is reachable or remove it.

** INV-4: Ungrounded Definitions (Math+Math)

Every scope node (definition, binding, theorem) in the math column should be
referenced by at least one =iatc= (argumentation) edge. Ungrounded definitions
are math-column islands — present but not connected to reasoning.

/Resolution:/ Add an iatc edge (=assert=, =reference=, =clarify=) linking the
scope to a post.

* The violation resolution loop

The full loop for any invariant:

1. *Detect:* =python3 scripts/ingest-three-columns.py --invariants=
2. *Browse:* =M-x arxana-browse= -> Violations -> click a violation
3. *Act:* Follow the resolution guidance in the detail buffer
4. *Resolve:* Re-run ingestion + invariants; violation count drops
5. *Evidence:* Both the original violation and the fix are durable hyperedges

This loop is the I4-compliant path from the AIF+ formalism: invariant rules
(slow timescale) are not rewritable by code changes (fast timescale) without
going through the project column (creating a tension, reviewing, recording).

* Extending with new invariants

To add a new invariant:

1. Add a check function to =scripts/ingest-three-columns.py= in the
   =check_invariants()= section
2. Use =query_all_of_type()= to fetch relevant hyperedges
3. Compute violations as a set difference or property check
4. Append violations to the violations list with invariant ID and resolution
5. Add the new =invariant/<type>= to the count function's type list
6. Add the type to =arxana-browser--violations-items= fetch list in
   =arxana-browser-lab.el=
7. Add face and context text for the new invariant type
