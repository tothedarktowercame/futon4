* Context
The evidence landscape stores every observation, claim, and coordination event
as a queryable entry. Evidence entries form chains via ~in-reply-to~ links,
creating navigable threads of reasoning.

~GET /api/alpha/evidence~ accepts filters: ~subject-type~, ~subject-id~, ~type~,
~claim-type~, ~tag~, ~author~, ~session-id~, ~since~, and ~limit~.

The evidence shape includes:

- ~evidence/subject~ — what this evidence is about (mission, thread, session)
- ~evidence/type~ — coordination, forum-post, mission-snapshot, portfolio-review
- ~evidence/claim-type~ — observation, goal, conclusion, conjecture
- ~evidence/tags~ — queryable tags for filtering
- ~evidence/body~ — the payload (chat text, portfolio snapshot, review data)

Chain navigation: ~GET /api/alpha/evidence/:id/chain~ returns the ordered
ancestor reply chain for any entry, following ~in-reply-to~ links backward.

Tag-based filtering: ~GET /api/alpha/evidence?tag=mission,backfill~ returns
entries tagged with both ~mission~ AND ~backfill~ (AND semantics). This is how
Arxana's narrative trail view finds all evidence for a specific mission.

* Implementation
- Query: ~evidence/backend.clj~ (all backends support ~:query/tags~)
- Chain: ~evidence/backend.clj~ (~fetch-chain~)
- Write: ~POST /api/alpha/evidence~ appends one entry
- Count: ~GET /api/alpha/evidence/count~ for efficient counting without payload
