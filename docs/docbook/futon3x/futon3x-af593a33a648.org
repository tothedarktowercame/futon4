* Context
The three-column stack (docbook://futon3x/futon3x-5b802804c69b) needs data
from three different sources. This entry documents the ingestion pipeline
that populates the futon1a hyperedge store.

* Prerequisites

- futon1a running on =:7071= with penholder =api= in =FUTON1A_ALLOWED_PENHOLDERS=
- futon3c running on =:7070= (for code reflection and project APIs)
- Math JSON files in =futon6/data/first-proof/=

* Usage

#+BEGIN_SRC sh
# Full ingestion: all three columns + invariant checks
python3 futon4/scripts/ingest-three-columns.py --all

# Individual columns
python3 futon4/scripts/ingest-three-columns.py --math
python3 futon4/scripts/ingest-three-columns.py --code
python3 futon4/scripts/ingest-three-columns.py --project

# Run invariant checks only (requires data already ingested)
python3 futon4/scripts/ingest-three-columns.py --invariants

# Report current hyperedge counts
python3 futon4/scripts/ingest-three-columns.py --count
#+END_SRC

* Data sources

** Math column (=--math=)

Source: =futon6/data/first-proof/thread-633512-hypergraph.json=

Each JSON node becomes a single-endpoint hyperedge (entity registration).
Each JSON edge becomes a multi-endpoint hyperedge (relation). Types are
prefixed with =math/= (e.g., =math/post=, =math/iatc=, =math/scope=).

Also ingests CT validation wires from =futon6/data/ct-validation/wires.json=
if present.

** Code column (=--code=)

Source: futon3c reflection API (=GET /api/alpha/reflect/*=)

Snapshots all futon-related namespaces (102 as of 2026-03-04), reflects
public vars for 54 key namespaces (peripherals, reflection, logic, transport,
API layers), and captures namespace dependency edges.

Types: =code/namespace=, =code/var=, =code/ns-contains-var=, =code/requires=.

** Project column (=--project=)

Source: futon3c Mission Control API (=GET /api/alpha/mc/*=)

Fetches devmaps, components, tensions, and trace paths. Each becomes a
hyperedge with =project/= prefix.

Types: =project/devmap=, =project/component=, =project/devmap-contains=,
=project/tension=, =project/tension-on=, =project/trace-path=.

* Idempotency

All writes use futon1a's stable ID scheme: =hx:{type}:{sorted-endpoints}=.
Re-running ingestion overwrites existing hyperedges with current data.
The pipeline is safe to run repeatedly.

* Hyperedge identity patterns

| Column | Entity pattern | Example |
|--------|---------------|---------|
| Math | ={node-id}= | =a-633527=, =a-633527:scope-000= |
| Code | =ns:{namespace}= or =var:{ns/sym}= | =ns:futon3c.reflection.core= |
| Project | =dm:{id}= or =comp:{dm/comp}= | =dm:peripheral-gauntlet= |
| Tension | =tension:{dm/comp}= | =tension:peripheral-gauntlet/C-arena= |
| Invariant | entity ID of the violating entity | =ns:futon3.gate.util= |

* Verification

After ingestion, verify round-trip by reading back individual hyperedges:

#+BEGIN_SRC sh
# Math
curl -s "http://localhost:7071/api/alpha/hyperedge/hx:math/post:a-633527"

# Code
curl -s "http://localhost:7071/api/alpha/hyperedge/hx:code/namespace:ns:futon3c.reflection.core"

# Project
curl -s "http://localhost:7071/api/alpha/hyperedge/hx:project/devmap:dm:peripheral-gauntlet"
#+END_SRC

All three should return the correct type, endpoints, and props.
